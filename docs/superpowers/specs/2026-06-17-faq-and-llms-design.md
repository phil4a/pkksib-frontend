# FAQ и llms.txt для PKKSib

## Цель

Добавить на сайт:

- отдельную SEO-страницу `/faq` со всеми вопросами и ответами;
- FAQ-блоки по 3-4 релевантных вопроса на страницах услуг и категорий услуг;
- текстовый маршрут `llms.txt` с кратким описанием компании и ссылками на ключевые разделы сайта.

Решение должно работать через SSR/SSG, чтобы контент был доступен поисковым роботам и LLM-краулерам без клиентского `fetch`.

## Контекст проекта

- Проект работает на Next.js App Router.
- Страницы услуг и категорий уже собираются в `src/app/services/[slug]/page.tsx`.
- `sitemap.xml` и `robots.txt` уже реализованы через metadata routes.
- Данные услуг приходят из Strapi, но новый FAQ в этой задаче не переносится в CMS.

## Границы задачи

Входит в задачу:

- создание единого реестра FAQ в коде;
- создание маршрута `/faq`;
- создание переиспользуемого accordion UI для FAQ;
- добавление JSON-LD schema для FAQ-страницы и страниц услуг/категорий;
- ручная привязка вопросов к `slug` услуг и категорий;
- создание маршрута `src/app/llms.txt/route.ts`;
- добавление `/faq` в sitemap.

Не входит в задачу:

- создание коллекции FAQ в Strapi;
- админский интерфейс для управления вопросами;
- автоматическая привязка вопросов по ключевым словам;
- изменение структуры существующих страниц услуг сверх интеграции FAQ-блока.

## Выбранный подход

Используется один статический источник данных в коде:

- `src/types/faq.types.ts` для типов;
- `src/data/faq.ts` для всех вопросов, категорий и карты соответствий `slug -> faq ids`;
- переиспользуемые UI-компоненты для вывода accordion и schema.

Почему выбран этот вариант:

- минимально затрагивает существующую архитектуру;
- даёт предсказуемый SSR/SSG;
- не требует изменений Strapi и API;
- позволяет быстро запускать FAQ и контролировать релевантность вопросов на каждой странице.

## Данные и структура

### Типы

Планируется ввести следующие сущности:

- `FaqItem`:
  - `id: string`
  - `question: string`
  - `answer: string`
  - `category: FaqCategoryKey`
- `FaqCategory`:
  - `key: FaqCategoryKey`
  - `title: string`
- `FaqSlugMap`:
  - словарь формата `Record<string, string[]>`, где ключом выступает `slug` услуги или категории, а значением массив `id` вопросов.

### Хранение контента

В `src/data/faq.ts` будут храниться:

- список категорий FAQ;
- полный массив `faqItems`;
- карта `faqSlugMap` для услуг и категорий;
- служебные функции:
  - получить все вопросы;
  - сгруппировать вопросы по категориям;
  - получить вопросы по `slug`;
  - ограничить локальные блоки до 3-4 записей.

### Правила привязки

- На странице `/faq` выводятся все вопросы, сгруппированные по категориям.
- На странице услуги или категории выводится подмножество вопросов по `slug`.
- Если для `slug` нет связей, FAQ-блок не рендерится.
- На локальных страницах показывается максимум 4 вопроса.

## UI и рендеринг

### Страница `/faq`

Маршрут: `src/app/faq/page.tsx`

Состав страницы:

- заголовок страницы;
- короткое вводное описание;
- секции по категориям;
- внутри каждой секции accordion со списком вопросов.

Требования:

- серверный рендеринг;
- семантические заголовки категорий;
- вопросы оформлены как accordion;
- ответы остаются в HTML страницы и не подгружаются отдельно.

### Accordion-компоненты

Переиспользуемые компоненты:

- `FaqSection` для секции категории;
- `FaqAccordion` для списка вопросов;
- `FaqBlock` для универсального блока, который можно вставлять и на `/faq`, и на страницы услуг.

Визуальные требования:

- каждый вопрос и ответ выглядят как accordion;
- компонент должен быть доступным с клавиатуры;
- используются стандартные HTML-элементы раскрытия или уже принятый в проекте простой client component;
- визуально блок вписывается в текущую стилистику сайта без отдельной дизайн-системы.

### Встраивание в страницы услуг и категорий

Точка интеграции: `src/app/services/[slug]/page.tsx`

Правило вывода:

- для категории FAQ-блок рендерится после основного контента списка услуг и текстового описания, перед формой;
- для услуги FAQ-блок рендерится после описания и связанных объектов, перед `EstimateForm`.

Причина:

- блок находится после основного коммерческого контента;
- сохраняется читаемая структура страницы;
- FAQ поддерживает SEO без вмешательства в hero-секцию.

## SEO и schema

### JSON-LD

Добавляется компонент `FaqSchema`, который на вход принимает массив `FaqItem[]` и генерирует schema.org `FAQPage`.

Правила:

- на странице `/faq` schema содержит все отображаемые вопросы;
- на страницах услуг и категорий schema содержит только реально показанные вопросы;
- в schema передаётся только текст ответа, без HTML.

### Sitemap

В `src/app/sitemap.ts` добавляется статическая страница `/faq`.

### Robots

`src/app/robots.ts` менять не требуется, так как маршрут уже разрешает индексацию всего сайта.

## llms.txt

Маршрут: `src/app/llms.txt/route.ts`

Формат ответа:

- `Content-Type: text/plain; charset=utf-8`;
- абсолютные URL на основные страницы сайта;
- краткая справка о компании;
- отдельные ссылки на `/faq`, `/prices`, `/services`, `/contacts`, `/about`.

Содержимое должно включать:

- название компании;
- специализацию;
- регион работы;
- ключевые направления;
- ссылки на важные разделы;
- краткое указание, что FAQ содержит ответы по кровле, фасадам и деревянному домостроению.

## Ошибки и отказоустойчивость

- Если `slug` не найден в карте FAQ, локальный блок не рендерится.
- Если массив вопросов пуст, schema не рендерится.
- Если в данных FAQ появится несуществующий `id`, служебная функция просто отфильтрует его при сборке списка.

## Тестирование и проверка

Минимальная проверка после реализации:

- `npm run lint`;
- ручная проверка страницы `/faq`;
- ручная проверка одной страницы услуги и одной страницы категории;
- проверка содержимого `/llms.txt`;
- проверка наличия `/faq` в `sitemap.xml`;
- проверка HTML на присутствие JSON-LD schema.

Автотесты в рамках задачи не обязательны, если реализация остаётся простой и полностью покрывается ручной проверкой и линтером.

## Ограничения и допущения

- FAQ остаётся статическим контентом в коде на текущем этапе.
- Привязка по `slug` поддерживается вручную.
- Вопросы и ответы берутся из `faq-pkksib.md`, при необходимости с редактурой в формат TypeScript-данных без изменения смысла.
- Для локальных FAQ-блоков допустимо использовать 3 или 4 вопроса в зависимости от релевантности страницы.

## Результат

После реализации сайт получает:

- полноценную SEO-страницу FAQ;
- тематические FAQ-блоки на страницах услуг и категорий;
- серверный `llms.txt` для LLM-краулеров;
- согласованную schema-разметку для видимого FAQ-контента.
